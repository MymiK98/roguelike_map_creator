# 로그라이크 맵/던전 생성 알고리즘 정리

> 절차적 던전 생성(Procedural Dungeon Generation, PDG) 프로그램 제작을 위한 주요 알고리즘의 원리·장단점·구현 방법과 출처 정리.
> 작성일: 2026-06-15 · 모든 핵심 주장은 다중 1차 출처와 3-0 만장일치 검증을 통과함.

---

## 0. 개요

로그라이크 맵 생성 기법은 크게 7가지 계열로 분류된다.

| # | 계열 | 핵심 특징 | 연결성 보장 | 결과물 형태 |
|---|------|-----------|------------|------------|
| 1 | **BSP 트리** (공간 분할) | 재귀 분할 후 리프에 방 배치 | 보장(복도 연결) | 구조적·격자형 던전 |
| 2 | **셀룰러 오토마타** | 노이즈 + 반복 스무딩 | 미보장 | 자연스러운 동굴 |
| 3 | **드렁큰/랜덤 워크** | 에이전트가 통로를 파냄 | 단일=보장 / 다중=미보장 | 유기적 동굴 |
| 4 | **룸-앤-코리더(터널링)** | 방 무작위 배치 + 직전 방 연결 | 보장(체인) | 방+복도 던전 |
| 5 | **미로 생성** | 신장 트리 기반 통로 | 보장 | 미로 |
| 6 | **그래프 기반** (들로네+MST) | 방 그래프 → MST 연결 | 보장 | 방+루프 던전 |
| 7 | **웨이브 함수 붕괴(WFC)** | 인접 규칙 제약 전파 | 미보장 | 타일 패턴 맵 |

> 실무에서는 단일 기법보다 **하이브리드 파이프라인**(예: BSP로 구획 → 일부 셀에 CA 동굴 → WFC로 디테일)이 권장된다. 특히 WFC는 단독 사용보다 다른 PDG와 결합 시 강력하다.

## 1. BSP(이진 공간 분할) 트리

### 원리
1. 벽으로 채운 직사각형 던전에서 시작.
2. 무작위 방향(수평/수직)·무작위 위치로 영역을 둘로 **재귀 분할**. 방 크기 정도의 하위 던전이 될 때까지 반복.
3. 각 **리프(leaf)** 노드에 무작위 크기 방을 배치.
4. **복도 연결**: 리프를 순회하며 형제(sister) 노드끼리 연결. 벽이 마주 보면 직선 복도, 아니면 Z자형 복도. 트리를 한 단계씩 올라가며 최상위 두 하위 던전이 이어질 때까지 반복.

### 장단점
- ✅ 방이 각자의 하위 던전 안에 포함되므로 **방 간 중첩이 구조적으로 방지**됨.
- ✅ 톱다운 맨션형 던전에 가장 흔히 쓰이는 기법.
- ⚠️ 복도 연결 방식에 변형 존재: **형제 노드 기반 계층 연결**(canonical) vs **리프 리스트 순차 연결**(SBGames 논문 변형). 단일 정답 아님.

- slsdo 구현 데모: https://slsdo.github.io/procedural-dungeon/
- SBGames 2020 비교 논문: https://www.sbgames.org/proceedings2020/ComputacaoShort/207911.pdf

## 2. 셀룰러 오토마타(CA) — 동굴 생성

### 원리
1. 초기 맵을 무작위로 채움(보통 바닥/벽 비율 **40~50%**).
2. 사용자 정의 규칙을 **n회 반복** 적용해 "스무딩".
3. 대표 **4-5 규칙**: 타일이 벽이고 8이웃 중 4개 이상이 벽이면 벽 유지 / 벽이 아니고 5개 이상이 벽이면 벽으로 전환 (= 3×3 영역에 벽 5개 이상이면 벽).
4. 매 반복마다 타일이 이웃과 비슷해져 노이즈가 점진 감소 → 자연스러운 동굴 형성.

### 변형/파라미터
- 이웃 정의: **무어**(직교+대각 8칸) 또는 **폰 노이만**(직교 4칸).
- 규칙이 많아질수록 느려져 성능에 큰 영향.
- slsdo 구현 예: "1칸 내 벽 5개↑ → 벽 / 2칸 내 벽 2개↓ → 벽" 규칙을 3회 적용 후 4회 정리 반복.

### 장단점
- ✅ 유기적·자연스러운 동굴.
- ❌ **연결성 미보장** → 사후 처리 필요(플러드필로 영역 식별 후 터널 연결).

### 출처
- RogueBasin — Cellular Automata Method: https://www.roguebasin.com/index.php/Cellular_Automata_Method_for_Generating_Random_Cave-Like_Levels
- SBGames 2020: https://www.sbgames.org/proceedings2020/ComputacaoShort/207911.pdf
- slsdo: https://slsdo.github.io/procedural-dungeon/
- marukrap 리소스 모음: https://github.com/marukrap/RoguelikeDevResources

## 3. 드렁큰 워크(랜덤 워크)

### 원리
- 본질적으로 랜덤 워크. 에이전트가 벽으로 채워진 맵을 **파내며** 통로 생성.
- 모든 에이전트를 **같은 위치**에서 시작 → 단일 동굴.
- **다른 위치**에서 시작 → 동굴 시스템(단, 영역 간 연결성 미보장).

### 장단점
- ✅ 구현이 매우 단순. **단일 워크는 완전 연결 보장**.
- ❌ 다중 에이전트 시 분리 영역 발생 가능.
- 동작이 구현마다 다름(스텝 수, 분기 확률 등).

### 출처
- SBGames 2020: https://www.sbgames.org/proceedings2020/ComputacaoShort/207911.pdf
- RogueBasin — Random Walk Cave Generation: https://www.roguebasin.com/index.php/Random_Walk_Cave_Generation
- PCG Wiki — Drunkard Walk: http://pcg.wikidot.com/pcg-algorithm:drunkard-walk

## 4. 룸-앤-코리더(터널링)

### 원리
1. 무작위 위치에 방을 잡음.
2. 기존 방과 겹치지 않으면 파냄.
3. **직전 방과 터널로 연결**.
4. 반복.

새 방을 직전 방에 잇는 방식은 경로 그래프(체인)를 형성 → **모든 방 도달 가능(완전 연결성 보장)**.

### 장단점
- ✅ 구현 단순, 완전 연결 보장. 입문 튜토리얼 표준.
- ⚠️ 터널끼리 또는 터널이 방을 관통하며 겹침. (플레이어는 터널/방 바닥을 구분 못 해 큰 문제는 아님.)

### 출처
- RogueBasin — Complete Roguelike Tutorial (libtcod, part 3): https://www.roguebasin.com/index.php/Complete_Roguelike_Tutorial,_using_python+libtcod,_part_3

### 변형: 랜덤 방 배치 + A* 복도
- 방 무작위 배치 후 **A\* 길찾기**로 문→문 최적 경로를 복도로 생성.
- 맨해튼 거리 휴리스틱 + 이진 힙 사용. 가중치로 복도가 직선이 되고 서로 합쳐지도록 유도.
- 출처: https://slsdo.github.io/procedural-dungeon/

## 5. 미로 생성 알고리즘

| 알고리즘 | 원리 | 특징 |
|----------|------|------|
| **재귀 백트래커** | 스택 기반 DFS "드렁큰 워크". 막다른 길에서 노드 pop | 길고 구불구불한 통로, 막다른 길 적음 |
| **Aldous-Broder / Wilson** | 균일(편향 없는) 신장 트리 생성 | 느림. Wilson이 더 빠르나 크기 커지면 여전히 느림 |
| **Kruskal / Prim** | MST 알고리즘 + 무작위 가중치 | Kruskal은 통로가 위/아래로 교차하는 **위브(weave) 미로** 가능 |

## 6. 그래프 기반 생성 (들로네 삼각분할 + MST)

### 원리 (TinyKeep 방식)
1. 방들을 무작위 배치.
2. 방 중심점으로 **들로네 삼각분할** → 인접 그래프 생성.
3. **최소 신장 트리(MST)** 추출 → 모든 방 연결 보장.
4. MST에서 제거된 엣지 중 **일부를 무작위로 다시 추가** → 루프/순환 경로 생성(단조로움 방지).

관련 변형: Relative Neighborhood Graph(상대 이웃 그래프).

### 출처
- marukrap — RoguelikeDevResources (Graph 카테고리: TinyKeep, Delaunay Triangulation Dungeon Generator, Relative Neighborhood Graph): https://github.com/marukrap/RoguelikeDevResources
- TinyKeep 알고리즘 설명(GameDev.net): https://www.gamedev.net/forums/topic/642621-my-procedural-dungeon-generation-algorithm-explained-for-my-game-tinykeep/

## 7. 웨이브 함수 붕괴(WFC)

### 원리
1. 작은 입력 이미지/타일맵에서 **초기 타일 모음 + 인접(adjacency) 규칙** 추출.
2. 각 셀의 가능한 패턴 집합을 점진적으로 축소.
3. 한 셀을 하나로 **붕괴(collapse)**시키고, 그 제약을 이웃으로 **전파**(스도쿠와 유사).
4. 결과: 입력과 **국소적으로 유사한** 더 큰 구조화 맵.

> 원래 샘플 이미지로부터의 **텍스처 합성**을 위해 고안됨.

### 장단점
- ✅ 사용자가 타일을 직접 디자인하는 **예술적·창의적 자유**. 기본 개념 단순.
- ❌ 동질성/대규모 구조 부재.
- ❌ 디테일 추가 시 과적합 → 다양성 저하.
- ❌ 영역 간 **연결성 미보장**.
- ❌ 높은 실행 비용 + 제약이 **지역적**이라 전역 결과 통제력 부족.
- 👉 **다른 PDG 알고리즘과 결합 권장.**

- CEIG 2025 논문: https://diglib.eg.org/bitstream/handle/10.2312/ceig20251107/ceig20251107.pdf

## 9. 선택 가이드 (요약)

- **구조적 던전(방+복도) 원함** → BSP 또는 룸-앤-코리더(터널링).
- **자연스러운 동굴 원함** → 셀룰러 오토마타 또는 드렁큰 워크 (+ 사후 연결성 처리).
- **방 간 흥미로운 루프/지름길 원함** → 그래프 기반(들로네+MST + 엣지 재추가).
- **미로/퍼즐 레벨** → 재귀 백트래커(빠름) 또는 Kruskal/Prim(위브).
- **타일 기반 양식화 맵** → WFC (다른 기법과 결합).

> ⚠️ 연결성 미보장 기법(CA, 다중 드렁큰 워크, WFC)은 반드시 **플러드필 영역 식별 → 가장 가까운 영역 터널 연결** 같은 사후 처리를 추가할 것.
